# Co:Efficient システム設計 (System Design)

## 1. システム概要 (System Overview)

Co:Efficientは、**ローカル-ファースト**なダッシュボードシステムで、以下で構成される：

- **メインデバイス** (Main Device): フロントエンド + バックエンド + データベース
- **サブデバイス** (Sub-Devices): フロントエンドのみ（メインデバイスのAPIに接続）

## 4. データフロー (Data Flow)

### 4.1 天気データフロー

```
┌─────────────────────────────────────────────────────────────┐
│ Frontend: useWeather Hook                                   │
└─────────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────┐
│ openMeteo.js: fetchWeatherData(forceRefresh)                │
│                                                             │
│ 1. if forceRefresh=false:                                   │
│    → getWeatherFromDB()                                     │
│    → if exist AND valid (≤16min): return                    │
│    → if not exist OR stale: fetch API                       │
│                                                             │
│ 2. if forceRefresh=true:                                    │
│    → always fetch API (sync time: 1,16,31,46min)            │
└─────────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────┐
│ Backend API: /api/weather/get                               │
│              /api/weather/save                              │
└─────────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────┐
│ Database: weather_data table                                │
│ - id                                                        │
│ - location_id                                               │
│ - timestamp (epoch time)                                    │
│ - data (JSON blob)                                          │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 天気データ取得ロジック

```
ページロード/リロード
    ↓
useWeather hook 実行
    ↓
fetchWeatherData(false) 呼び出し
    ↓
┌─ DBにデータあり? ──No──> API呼び出し → DB保存 → 終了
│
└─ Yes 
    ↓
   データ有効? (≤16分古い)
    ├─ No  →  API呼び出し   →  DB保存 → 終了
    └─ Yes →  DB データ使用 →  終了

バックグラウンドプロセス (毎分チェック)
    ↓
同期時刻? (1,16,31,46分)
├─ Yes → API呼び出し → DB保存 → UI更新
└─ No  → 待機
```

## 6. データベーススキーマ (Database Schema)

### 6.1 天気データ テーブル

```sql
CREATE TABLE weather_data (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    location_id INTEGER NOT NULL,
    timestamp INTEGER NOT NULL,           -- json imbeded timestamp
    data JSON NOT NULL,                   -- Open-Meteo 完全レスポンス
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

### 6.2 アナウンスメント テーブル

```sql
CREATE TABLE announcements (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    title VARCHAR(150) NOT NULL,
    subtitle VARCHAR(255),
    content TEXT NOT NULL,
    description TEXT,
    tags JSON,                            -- [{id, name, color, order}]
    created_by INTEGER NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    modified_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

### 6.3 スケジュール テーブル

```sql
CREATE TABLE schedules (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    title VARCHAR(150) NOT NULL,
    subtitle VARCHAR(255),
    description TEXT,
    tags JSON,                            -- [{id, name, color}]
    time_start TIMESTAMP,
    time_end TIMESTAMP,
    created_by INTEGER NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    modified_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    is_finished BOOLEAN DEFAULT FALSE
);
```

### 6.4 ロケーション テーブル

```sql
CREATE TABLE locations (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name VARCHAR(100) NOT NULL,
    longitude FLOAT NOT NULL,
    latitude FLOAT NOT NULL
);
```

## 9. キャッシング戦略 (Caching Strategy)

### 現在の方針 (Current Approach: v0.1.0)

**キャッシュレス・DB-ファースト戦略**

すべてのデータをSQLiteデータベースに保存し、キャッシュレイヤーは使用しない。

| データ | 保存場所 | TTL | 検証方法 |
| --- | --- | --- | --- |
| 天気データ (Raw JSON) | SQLite `weather_data` テーブル | なし | タイムスタンプ比較 (≤16分) |
| 天気コンパイル済みデータ | メモリ (React State) | - | オンデマンド計算 |
| UI状態 | React State | - | コンポーネント内 |
| ユーザー設定 | SQLite | なし | 永続的 |
| アナウンスメント | SQLite | なし | 永続的 |
| スケジュール | SQLite | なし | 永続的 |

### 天気データ キャッシュ検証フロー

```markdown
ページロード
  ↓
DB から最新データ取得
  ↓
タイムスタンプ確認
  ├─ データなし　　　 →  API呼び出し   → DB保存
  ├─ 古い (>16分)    →  API呼び出し   → DB保存
  └─ 新しい (≤16分)  →  DB データ使用
バックグラウンド (同期時刻: 1,16,31,46分)
  ↓
強制更新 → API呼び出し → DB保存 → UI更新
```

## 10. エラーハンドリング (Error Handling)

### 10.1 API エラー

```jsx
HTTP Status Code | 処理
400              | クライアントエラー (入力検証)
401              | 認証失敗
403              | 権限不足
404              | リソース見つからない
500              | サーバーエラー (再試行)
```

### 10.2 フロントエンド エラー状態

```
Loading状態
    ↓
Error発生?
├─ Yes:
│   ├─ API Error: エラーメッセージ表示
│   ├─ Network Error: 再試行ボタン表示
│   └─ Timeout: キャッシュデータ使用
└─ No: データ表示
```

## 11. パフォーマンス最適化 (Performance Optimization)

### 11.1 フロントエンド

- コンポーネント分割 (Weather, Forecast, Announcements)
- React.memo で不要な再描画を防止
- useCallback で関数再生成を防止
- 遅延ローディング (Lazy Loading)

### 11.2 バックエンド

- データベースインデックス
- キャッシュ戦略
- クエリ最適化
- API レスポンス圧縮

### 11.3 ネットワーク

- API呼び出し最小化 (15分ごと)
- JSON データ最適化
- gzip圧縮
