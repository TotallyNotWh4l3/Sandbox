# Co:Efficient

# 📘 ——背景——

> 
> 
> 
> 天気情報や交通情報を提供するサービスはすでに存在している。
> 
> しかし、それらに加えて連絡事項や予定なども含めた情報を把握するには、**複数のアプリやサービス**を行き来する必要があり、情報取得までの**ステップが多い**。
> また、既存の天気サービスの多くは**UIデザインが見づらく**、重要な情報とそうでない**情報の優先順位**が視覚的に伝わりにくい。
> 
> さらに、機能的なオフィスや施設は多いが、**視覚的には単調**になりがちである。
> 情報を表示するディスプレイが一つあるだけで、**空間に個性と雰囲気**が生まれる。
> 
> **Co:Efficient** はその可能性にも着目している。
> 

---

# 📘 ——目的——

> **Co:Efficient** は、オフィス・学校・施設などで働く人々が、天気・注意報・連絡事項・予定などの情報を一目で把握できる、モジュール式の常時表示型ダッシュボードシステムである。情報取得までのステップを最小化し、実用性と視覚的な美しさを兼ね備えた空間づくりに貢献することを目的とする。完全ローカル動作・無料データソースのみを使用し、サーバー不要・サブスクリプション不要のオープンソースアプリケーションとして公開する。
> 

---

# 📘 ——目標——

> 使用者の **70%** が情報を **3秒以内** に把握できるよう目指す。
※「全くそう思わない」「そう思わない」「どちらでもない」「そう思う」「強くそう思う」の５段階評価
次のアンケートの質問に回答者の **70%** が 「そう思う」または、「強くそう思う」っと答えた。
> 
> - 「読みやすく、知りたい情報がすぐに把握できた」
> - 「説明なしでも、どこに何の情報があるかわかった」
> - 「このダッシュボードは実際のオフィスや学校で役立つと思う」

## 4. データフロー (Data Flow)

### 4.1 天気データフロー

```
┌─────────────────────────────────────────────────────────────┐
│ Frontend: useWeather Hook                                   │
└─────────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────┐
│ openMeteo.js: fetchWeatherData(forceRefresh)                │
│                                                             │
│ 1. if forceRefresh=false:                                   │
│    → getWeatherFromDB()                                     │
│    → if exist AND valid (≤16min): return                    │
│    → if not exist OR stale: fetch API                       │
│                                                             │
│ 2. if forceRefresh=true:                                    │
│    → always fetch API (sync time: 1,16,31,46min)            │
└─────────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────┐
│ Backend API: /api/weather/get                               │
│              /api/weather/save                              │
└─────────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────┐
│ Database: weather_data table                                │
│ - id                                                        │
│ - location_id                                               │
│ - timestamp (epoch time)                                    │
│ - data (JSON blob)                                          │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 天気データ取得ロジック

```
ページロード/リロード
    ↓
useWeather hook 実行
    ↓
fetchWeatherData(false) 呼び出し
    ↓
┌─ DBにデータあり? ──No──> API呼び出し → DB保存 → 終了
│
└─ Yes 
    ↓
   データ有効? (≤16分古い)
    ├─ No  →  API呼び出し   →  DB保存 → 終了
    └─ Yes →  DB データ使用 →  終了

バックグラウンドプロセス (毎分チェック)
    ↓
同期時刻? (1,16,31,46分)
├─ Yes → API呼び出し → DB保存 → UI更新
└─ No  → 待機
```

## 9. キャッシング戦略 (Caching Strategy)

### 現在の方針 (Current Approach: v0.1.0)

**キャッシュレス・DB-ファースト戦略**

すべてのデータをSQLiteデータベースに保存し、キャッシュレイヤーは使用しない。

| データ | 保存場所 | TTL | 検証方法 |
| --- | --- | --- | --- |
| 天気データ (Raw JSON) | SQLite `weather_data` テーブル | なし | タイムスタンプ比較 (≤16分) |
| 天気コンパイル済みデータ | メモリ (React State) | - | オンデマンド計算 |
| UI状態 | React State | - | コンポーネント内 |
| ユーザー設定 | SQLite | なし | 永続的 |
| アナウンスメント | SQLite | なし | 永続的 |
| スケジュール | SQLite | なし | 永続的 |

```markdown
ページロード
  ↓

  ↓
タイムスタンプ確認
  ├─ データなし　　　 →  API呼び出し   → DB保存
  ├─ 古い (>16分)    →  API呼び出し   → DB保存
  └─ 新しい (≤16分)  →  DB データ使用
バックグラウンド (同期時刻: 1,16,31,46分)
  ↓
強制更新 → API呼び出し → DB保存 → UI更新
```
